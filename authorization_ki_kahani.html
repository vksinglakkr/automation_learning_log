Authorization ki Kahani Header, Bearer aur API Key — asal me farak kya hai?

Na koi PhD wali bhasha, na koi jaldi. Bas ek kahani, kuch tasveerein, aur end tak tum khud decide kar paoge ki kisi bhi naye API me apni key kahan likhni hai.

Chapter 1

Parcel wali kahaniThe parcel story — jahan se sab shuru hota hai

Raju Kurukshetra ke post office se ek parcel Delhi bhejna chahta hai. Us parcel ke saath teen tarah ki jagah hoti hain jahan wo cheezein likh sakta hai:

  1. Bahar ka address label — sabko dikhta hai, koi bhi padh sakta hai. (Yeh URL / query parameter jaisa hai.)
  2. Ek chhota sila hua envelope, parcel ke upar staple kiya hua — isme alag-alag khaane (boxes) bane hote hain jaise "Prapak" (To), "Preshak" (From), "Anumati Patra" (Authorization Slip). Postman inhi khaano ko dekhta hai. (Yeh HTTP headers jaisa hai.)
  3. Parcel ke andar ka asli saaman — jo cheez actually bhej rahe hain. (Yeh request body jaisa hai.)

Server (jaise data.gov.in ya Groq ka server) bhi bilkul isi tarah kaam karta hai — jab tumhara code koi request bhejta hai, usme ye teeno jagah maujood hoti hain, aur har company khud decide karti hai ki wo apni "chaabi" (key) kahan dhoondegi.

Chapter 2

"Authorization" hi kyun likhte hain?Kya koi aur naam likh sakte hain?

Yeh sabse zyada confuse karne wala sawal hai — haan, bilkul kisi bhi naam ka apna khaana (custom header) bana sakte ho. HTTP tumhe rokta nahi. Tum chahte to likh sakte the:

  • x-api-key: abc123
  • apikey: abc123
  • meri-chaabi: abc123 (bilkul, koi bhi naam)

Lekin "Authorization" ek pehle se tay (reserved / standard) naam hai — jaise sarkari form me pehle se chhape hue khaane hote hain "Naam", "Pita ka Naam", "Pata". Har server, browser, security tool — sabko pata hai ki agar kisi request me permission ka proof dhoondhna hai, to pehle Authorization naam ka khaana check karo. Isliye zyadatar companies (Groq, OpenAI, GitHub, bank APIs) yehi standard naam use karti hain.

Par yaad hai humara data.gov.in wala project? Wahan hum Authorization header bhejte hain — aur server use bilkul ignore kar deta hai! Kyunki data.gov.in ke engineers ne apna server aisa banaya hi nahi ki wo header dekhe. Unhone decide kiya ki key sirf address label (query parameter) me hi likhi jaani chahiye: ?api-key=abc123. Yehi wajah hai ki humare workflow me header_auth aur bearer_auth branch FAILS hoti hain — sahi jagah likha hi nahi, isliye rejected.

Naam kaha likhaKya likhte hainKaun use karta hai (example)
Header: AuthorizationBearer <key>Groq, OpenAI, GitHub
Header: custom naamsirf <key>, koi scheme word nahiKai payment / SaaS APIs, e.g. x-api-key

Golden rule: jo company API bana rahi hai, wahi decide karti hai key kahan dhoondhi jaayegi. Hum sirf unke documentation me likha hua follow karte hain — guess nahi karte.

Chapter 3

"Bearer" kya hota hai?Yeh header ke andar kyun likha jaata hai?

Socho tumhare paas bus ka ticket hai. Conductor ye nahi poochta "tumhara ID card dikhao" — wo bas ye dekhta hai ki tumhare paas ticket hai ya nahi. Jiske paas ticket hai, wahi safar kar sakta hai — chahe wo kisi ne bhi diya ho. Isko English me kehte hain: "bearer" — matlab "leke chalne wala / dikhane wala".

Bilkul waise hi, Authorization: Bearer abc123 ka matlab hai: "Jo bhi ye abc123 chit (token) leke aaya hai, use andar aane do — extra check ki zaroorat nahi."

Authorization header ke andar aksar do tukde hote hain: pehla scheme word (yeh bataata hai key ko kaise padhna hai), aur doosra asli key/token.

Scheme wordMatlabExample
Bearerbas token dikhao, koi extra check nahiAuthorization: Bearer sk-abc123

(kuch nahi)seedha raw key, koi scheme word nahix-api-key: abc123

Har API alag choose karti hai — koi universal rule nahi hai ki hamesha "Bearer" likhna hi hoga. Isiliye har baar naye API ke documentation page par jaake dekhna padta hai.

Chapter 5

Koi bhi naya API mile to...5 sawal jo hamesha poochne hain

  • 1. Key kahan jaati hai — URL me, header me, ya body me?Where does the key go?
  • 2. Agar header me, header ka naam kya hai — Authorization ya kuch custom?If header, which header name exactly?
  • 3. Agar Authorization hai, to koi scheme word (Bearer / Basic) chahiye ya sirf raw key?Does it need a scheme word before the key?
  • 4. Key ka format kya hai — plain text, base64, ya JWT jaisa token?What format is the key in?
  • 5. Kya har request ke saath key bhejni hai, ya sirf ek baar login karke session milta hai?Sent every time, or only once at login?

Yeh paanch sawal poochoge to koi bhi naya API — chahe Groq ho, chahe koi government portal, chahe koi payment gateway — sabme apni key sahi jagah laga paoge, bina kisi ki madad ke.

Blackboard Recap

Header
Request ke saath jud'a chhota label/khaana, jisme extra jaankari likhi jaati hai — jaise parcel par staple kiya hua envelope.
Authorization (header ka naam)
Ek standard, sabko-pata naam jisme permission ka proof likha jaata hai. Chahe to koi aur naam bhi bana sakte ho, par ye sabse common hai.
Bearer (scheme word)
Authorization header ki value ke andar likha jaane wala shabd — matlab "jiske paas ye token hai, wahi allowed hai", bus-ticket jaisa.
API Key
Asli chaabi/token jo tumhe pehchaanti hai. Ye kahin bhi ja sakti hai — URL me, header me, ya body me — depends sirf us API par.
Query Parameter
URL ke andar ?key=value jaisa likha hua data — sabko dikhta hai, jaise parcel ka bahar wala address label.
